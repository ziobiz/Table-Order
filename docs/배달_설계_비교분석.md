# 배달 기사·배차 설계 — 실제 배달앱 비교 분석

개발 초보자도 이해할 수 있도록, **실제로 운영 중인 배달앱(배민·쿠팡이츠·Uber Eats 등)**이 어떻게 돌아가는지 먼저 정리하고, 우리 시스템을 그에 맞춰 설계합니다.

---

## 1. 실제 배달앱은 어떻게 돌아가나요?

### 1-1. 고객이 보는 흐름 (간단히)

```
고객이 앱에서 주문 → 결제 → "배달 중" 화면에서 기사 위치 확인 → 도착 알림 → 수령
```

- 고객은 **주문 → 결제 → 배달 중(지도)·도착** 정도만 보면 됩니다.

### 1-2. 배달앱(플랫폼)이 하는 일

| 단계 | 설명 | 비고 |
|------|------|------|
| 주문 접수 | 가맹점에서 주문 들어옴. "배달 주문 1건 생김" 상태. | 우리: `orders` (order_type=delivery) |
| 배차 대기 | "누가 이 주문을 배달할지" 아직 정해지지 않음. | 우리: `deliveries.status = WAITING` |
| 기사 배정 | **방식 A** 플랫폼이 기사에게 자동/수동으로 넣어줌. **방식 B** 기사가 앱에서 "이 콜 받을게요" 하고 수락. | 아래에서 자세히 |
| 픽업 | 기사가 매장 도착 → 음식 수령 완료. | 픽업 완료 버튼 |
| 배달 중 | 기사가 고객 주소로 이동 중. | 상태만 "배달 중" |
| 배달 완료 | 고객에게 전달 완료. 기사가 "완료" 버튼. | 완료 처리 |

- 실제 앱들은 **"주문 접수 → 배차 → 픽업 → 배달 중 → 완료"** 이 순서를 그대로 따릅니다. 우리도 이 순서를 그대로 쓰면 됩니다.

### 1-3. 배달앱에서 "기사"는 누구인가?

- **플랫폼(본사)이 채용·등록한 기사**  
  - 배민·쿠팡이츠·Uber Eats처럼 **앱 하나로 여러 가맹점** 배달을 하는 사람.
  - **어느 매장이든** 반경 안에 있으면 콜이 뜨고, **기사가 직접 "수락"** 하거나, **알고리즘으로 자동 배정**됩니다.
  - 정산은 **건당 수수료**를 **플랫폼(본사)** 이 나중에 지급.

우리 시스템에서는 이걸 **"본사 소속 기사(HQ)"** 로 두면 됩니다.

- **가맹점이 직접 채용한 기사**  
  - 중국집·피자집처럼 **그 매장 전용**으로 일하는 사람.
  - **우리 매장 주문만** 배달하고, **점주/매장 담당자가 "이 기사한테 이 주문 맡겨"** 하고 지정합니다.
  - 급여는 **시급/월급**으로 **가맹점주**가 직접 지급.

우리 시스템에서는 이걸 **"가맹점 전용 기사(STORE)"** 로 두면 됩니다.

---

## 2. 실제 배달앱 vs 우리 시스템 비교

### 2-1. "배달앱처럼" = 본사 소속 기사(HQ)

| 비교 항목 | 배민·쿠팡이츠·Uber Eats | 우리 시스템 (HQ) |
|-----------|-------------------------|-------------------|
| 기사 소속 | 플랫폼(앱 회사) | **본사** (driver_type = HQ) |
| 활동 범위 | 여러 가맹점, 지역 기준 | **store_id = NULL** (어느 매장이든 가능) |
| 배차 방식 | 알고리즘 자동 배정 또는 기사가 콜 수락 | **AUTO**: 기사가 "대기 중인 주문 목록"에서 **직접 수락(accept)** |
| 기사 앱 화면 | 반경 N km 내 주문 목록/지도, "수락" 버튼 | **우리도**: 대기 중인 배달 목록(또는 지도) + "수락" 버튼 |
| 정산 | 건당 수수료, 플랫폼이 정산 후 지급 | **delivery_fee** 저장 → 본사 정산용 |

- 즉, **HQ = "배달앱과 같은 조건·로직"** 입니다.  
  - 주문 1건 = 배달 1건(`deliveries` 1행).  
  - 상태: **WAITING → (기사 수락) → ACCEPTED → PICKED_UP → DELIVERED**.

### 2-2. "가맹점 직원" = 가맹점 전용 기사(STORE)

| 비교 항목 | 중국집·피자집 직원 배달 | 우리 시스템 (STORE) |
|-----------|-------------------------|----------------------|
| 기사 소속 | 그 매장만 | **가맹점** (driver_type = STORE, store_id = 해당 매장) |
| 활동 범위 | 우리 매장 주문만 | **그 매장 주문만** (store_id 일치하는 주문만) |
| 배차 방식 | 점주가 "너 이거 배달해" 하고 지정 | **MANUAL**: 가맹점(점주)이 **"이 주문 → 이 기사"** 지정(assign) |
| 점주 화면 | 전화/메모로 부르거나, 간단 대기열 | **매장용 배차 대시보드**: "대기 중인 배달" + "우리 기사 목록" → 클릭해서 배정 |
| 정산 | 시급/월급, 점주가 직접 | 우리 시스템에서는 "배차·상태"만 관리, 급여는 별도 |

- 즉, **STORE = "가맹점이 직접 등록·채용하고, 점주가 수동으로 배차"** 하는 형태입니다.

## 3. 데이터가 어떻게 흐르는지 (쉽게)

### 3-1. 공통: "배달 1건"이 생기는 시점

- 고객이 **배달 주문** 하고 결제까지 완료 → `orders` 에 1건 (order_type = delivery, store_id = 매장).
- 이 주문에 대해 **배달 건** 1개를 만듦 → `deliveries` 에 1행 (order_id, status = WAITING, driver_id = NULL).

→ **실제 배달앱도 "주문 1건 = 배달 1건"** 으로 관리합니다. 우리도 동일하게 가져가면 됩니다.

### 3-2. 본사 기사(HQ) — "배달앱처럼"

1. 본사가 **기사 등록** → `drivers` (driver_type = HQ, store_id = NULL).
2. **대기 중인 배달** 조회 → `deliveries` 중 status = WAITING (필요하면 반경/지역 조건 추가).
3. 기사가 앱에서 **"이 배달 받을게요"** → `POST /api/deliveries/:id/accept`.  
   - 검증: 이 기사가 HQ인지, 이 배달이 아직 WAITING인지.  
   - 동시에 두 기사가 누르면 한 명만 성공하도록 **트랜잭션/락** 처리.
4. 성공 시: `deliveries.driver_id = 이 기사`, `status = ACCEPTED`, `dispatch_type = AUTO`.
5. 이후: 픽업 완료 → PICKED_UP, 배달 완료 → DELIVERED. (실제 앱과 동일한 단계.)

### 3-3. 가맹점 기사(STORE) — "점주가 지정"

1. 가맹점이 **기사 등록** → `drivers` (driver_type = STORE, store_id = 이 매장).
2. **우리 매장** "배달 대기 목록" 조회 → `deliveries` 중 WAITING 이고, 주문의 store_id = 우리 매장.
3. 점주가 **"이 배달 → 이 기사"** 지정 → `POST /api/deliveries/:id/assign` (driver_id, store_id 등 전달).  
   - 검증:  
     - 지정하려는 기사가 **STORE** 인지.  
     - 그 기사의 **store_id** 가 **이 배달의 주문(orders.store_id)** 과 **같은지**.  
   → 다른 매장 기사에게 배차되는 실수를 막습니다.
4. 성공 시: `deliveries.driver_id = 지정한 기사`, `status = ACCEPTED`, `dispatch_type = MANUAL`.
5. 이후: 픽업/배달 완료는 HQ와 동일 (PICKED_UP → DELIVERED).

## 4. DB 구조 (실제 앱과 맞춘 요약)

### 4-1. 기사 한 테이블로 두 타입 — `drivers`

- **실제**: 배민도 "라이더", 쿠팡이츠도 "라이더" 한 종류지만, 플랫폼 소속.  
  우리는 **플랫폼(HQ)** / **가맹점(STORE)** 두 종류를 **한 테이블**에서 `driver_type`으로 구분.
- **drivers**
  - `driver_type`: 'HQ' | 'STORE'
  - `store_id`: STORE일 때만 해당 매장 ID, HQ일 때는 NULL
  - 그 외: name, phone, username, password, is_active 등 (로그인·정산용은 기존처럼)

→ "배달앱 같은 기사"와 "가맹점 전용 기사"를 **같은 테이블에서 타입만 나누는** 구조라서, 나중에 통계·정산도 하나로 다루기 쉽습니다.

### 4-2. 배달 현황 한 테이블 — `deliveries`

- **실제**: 주문 1건당 "배달 1건"으로 상태만 WAITING → ACCEPTED → PICKED_UP → DELIVERED 로 관리.
- **deliveries**
  - `order_id`: 어떤 주문의 배달인지 (1:1)
  - `driver_id`: 누가 배달하는지 (배차 전엔 NULL)
  - `dispatch_type`: AUTO(기사가 수락) vs MANUAL(점주가 지정)
  - `status`: WAITING → ACCEPTED → PICKED_UP → DELIVERED
  - `delivery_fee`: 건당 배달비(본사 정산 등에 사용)

→ 실제 배달앱의 "배달 건 단위 상태 관리"와 동일한 방식입니다.

## 5. API·화면을 실제 앱에 맞추기

### 5-1. 가맹점 전용 기사(STORE) — 수동 배차

- **API**: `POST /api/deliveries/:id/assign`  
  - 가맹점(store_id)이 "이 배달(id)을 이 기사(driver_id)에게 맡긴다".  
  - 검증: 기사가 STORE + 기사 store_id = 이 배달의 주문 store_id.  
  - 성공 시: driver_id 반영, status = ACCEPTED, dispatch_type = MANUAL.
- **화면**: **매장용 배차 대시보드**  
  - "대기 중인 배달" 목록 + "우리 매장 기사" 목록.  
  - 점주가 "이 배달 → 이 기사" 클릭/선택해서 배정. (실제로는 "우리 직원 클릭해서 배치"하는 그 UI.)

### 5-2. 본사 소속 기사(HQ) — 콜 수락(배달앱처럼)

- **API**  
  - `GET /api/deliveries/available`: WAITING인 배달 목록 (나중에 반경/지역 필터 추가 가능).  
  - `POST /api/deliveries/:id/accept`: "이 배달 받을게요."  
    - 검증: 기사가 HQ, 배달이 WAITING.  
    - 동시성: 트랜잭션 + "이미 driver_id가 채워졌는지" 한 번 더 확인해서 한 명만 성공.  
  - 성공 시: driver_id 반영, status = ACCEPTED, dispatch_type = AUTO.
- **화면**: **기사 앱(목록/지도)**  
  - "반경 N km 내 대기 주문" 목록(또는 지도).  
  - 각 건마다 "수락" 버튼. (실제 배달앱과 같은 사용 방식.)

## 6. 정리 — "실제 배달앱과 비교해서 설계한 점"

1. **주문 1건 = 배달 1건**  
   - 실제 앱처럼 `orders`(주문) + `deliveries`(배달 건, 상태·기사·배달비) 로 나눔.
2. **기사는 한 테이블, 타입만 구분**  
   - `drivers` 에서 `driver_type`(HQ/STORE), `store_id`(STORE만) 로 "배달앱 같은 기사" vs "가맹점 전용 기사" 구분.
3. **배차 방식 두 가지**  
   - **AUTO**: HQ 기사가 대기 목록에서 직접 수락 (배민·쿠팡이츠처럼).  
   - **MANUAL**: 가맹점이 대기 중인 배달에 기사를 지정 (중국집·피자집처럼).
4. **상태 흐름은 실제 앱과 동일**  
   - WAITING → ACCEPTED → PICKED_UP → DELIVERED.
5. **검증으로 실수 방지**  
   - STORE 배차 시: "이 기사가 우리 매장 소속인지" 반드시 확인.  
   - HQ 수락 시: "이 배달이 아직 대기인지" + 동시 수락 방지.

이렇게 하면 **기존에 운영 중인 배달앱과 비교·분석해서 설계**한 구조가 되고,  
나중에 "배민은 이렇게 하는데 우리는?" 같은 질문에도 같은 기준으로 답할 수 있습니다.
